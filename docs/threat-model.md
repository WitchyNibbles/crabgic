# Threat model v1 — Engineering Orchestrator

**Status:** v1, produced as roadmap phase 02 work item 9.
**Scope:** STRIDE analysis over the nine surfaces named by `roadmap/02-contracts-and-schemas.md`'s exit
criterion: UDS (Unix domain socket control plane), worker runtime, envelope compiler, installer, gateway,
connectors, capability quarantine, renderer, learning store.
**Design baseline:** `docs/claude-code-adaptation.md` (verified against Claude Code 2.1.207, 2026-07-12) and
`docs/engine-baseline.md` (phase 00's live-verified facts, Claude Code 2.1.207–2.1.210, dated 2026-07-15).
Every engine-behavior claim below cites `docs/engine-baseline.md`, never memory, per the project's ground
rule. Every design/ownership claim cites the owning roadmap phase file.
**Author/process:** produced as an orchestrated documentation pass over the settled phase 02–23 design; no
new design decisions are made here — this document records threats against, and cites mitigations already
specified in, the existing roadmap and adaptation corpus. Where a phase's own text already flags an open
risk, that risk is carried into the relevant STRIDE cell rather than re-litigated.

## Purpose and method

Each of the nine surfaces gets a short description of its design (citing the adaptation doc section and/or
the owning roadmap phase), followed by a STRIDE table. Every table row names a concrete threat scenario
specific to this system's design (not a generic STRIDE textbook example), the mitigation that exists or is
planned — citing the exact owning phase and mechanism — and the residual risk that mitigation does not fully
close. A cell with "none material" as its residual risk means the reviewer found no design-specific gap worth
tracking beyond ordinary implementation-quality risk; every other cell names something concrete.

## Scope and non-goals

This is a **design-level** threat model. Phases 03, 05, 06, 10, 12, 16, 17, and 22 are unimplemented as of
this writing (only phase 02's contracts and phase 00/01's spikes/bootstrap exist in `packages/`); the
mitigations cited below are the specification each phase's own exit criteria will hold it to, not yet
verified against running code. Re-review is required once each phase lands (23's release-hardening pass is
the roadmap's designated re-verification point, per `02-contracts-and-schemas.md`'s own note that this
document is "consumed by 23: security review pass vs. 03/16/17 implementation"). This document does not
re-derive engine facts independently; every engine-behavior claim traces to a specific `docs/engine-baseline.md`
section, and every claim marked `UNRESOLVED` there is treated as unresolved here too, never as settled.

## Cross-cutting mitigations this phase (02) itself ships

Several controls referenced repeatedly below are not surface-specific — they are contracts phase 02 produces
that every surface built afterward inherits for free:

- **Redacting connector-error constructors** — the 10-member canonical connector-error union
  (`authentication, permission, not_found, conflict, rate_limited, validation, unsupported, transient,
ambiguous_write, policy_blocked`) has constructors that force provider-body redaction; "no raw body field
  exists on the public type," proven at the type level, not just at runtime (`02-contracts-and-schemas.md`
  work item 4, exit criteria). Every surface that touches a remote provider (gateway, connectors) inherits
  this by construction.
- **Security-monotonic config precedence resolver** — CLI → env → project → user → defaults, with a declared
  security-key set where lower precedence only ever tightens (deny-lists append-only, booleans one-way,
  numeric limits min-wins), proven by a ≥10k-case fast-check adversarial corpus that no config-layer stack
  can loosen a declared security key (`02-contracts-and-schemas.md` work item 8, exit criteria). The
  envelope compiler (03) and installer (10) both build layered, monotonicity-dependent behavior on top of
  this precedent even though 10 implements its own installer-specific merge writer rather than importing 02's
  resolver directly.
- **`GATEWAY_MCP_SERVER_NAME` single-definition-site constant** (`"eo_gateway"`) — every engine-side MCP
  registration (envelope compiler's allow-string, worker runtime's `mcpServers` key, installer's `.mcp.json`
  entry, gateway's own server registration) derives from this one exported constant; a repo-wide
  grep/golden-value CI check fails if the literal is hand-typed a second time anywhere under `packages/*`
  (`02-contracts-and-schemas.md` exit criteria). This closes an entire class of "the gateway allowlist and the
  spawn config disagree on the server name" spoofing/misconfiguration risk structurally, not procedurally.
- **`CommunicationPolicy` prohibited-content categories** (attribution, first-person, signatures, mentions,
  secrets, unsafe links) — the renderer (17) and every connector (18/19/20) read these limits programmatically
  at call time, never hardcoded, so a policy change can't silently desync a golden fixture without failing CI.
- **`renderer-core` module** (length/line counters + attribution-token scanner primitives, inside
  `packages/contracts`, not a standalone package) — the shared primitive the renderer's `lint()` stages and
  phase 08's belt-and-suspenders attribution assertion both call into, so the two surfaces never drift on what
  counts as an attribution leak.

## 1. UDS (Unix domain socket control plane)

**Design:** the supervisor daemon (`05-supervisor-daemon.md`) exposes an ndjson request/response + server-push
protocol over a Unix domain socket, mode `0600` inside a `0700` runtime dir, nested under 04's pinned
`$XDG_STATE_HOME/engineering-orchestrator/<project-hash>/` root. `SO_PEERCRED` uid-check is the trust
boundary; a versioned handshake rejects a mismatched protocol version before serving any request. Exactly two
local peer categories are admitted, both running as the invoking uid: the CLI (09) directly, and the gateway
(16), which forwards its own MCP-visible `run.status`/`run.cancel` tools over this same socket rather than
reimplementing them — "one handler set, two transports, never a second implementation" (05 §Goal).

| STRIDE                 | Threat scenario                                                                                                                                                                                 | Mitigation (owning phase + mechanism)                                                                                                                                                                                                                                                                                                                              | Residual risk                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Spoofing               | A same-uid local process opens the socket and impersonates the CLI or the gateway to issue `run.cancel` or read run/worker registries.                                                          | `SO_PEERCRED` uid-extraction check refuses any foreign-uid connection before serving a request (05 §UDS control plane; property-tested over randomized peer-uid sequences, 05 §Property test plan).                                                                                                                                                                | Trust is scoped to the OS user account, not to the CLI/gateway pairing specifically — any other same-uid process (including an unrelated tool run by the same user) also clears `SO_PEERCRED` and reaches the identical router. This is a stated design choice ("trusted by exactly two local peers sharing the invoking uid," 05 §Goal), not an oversight, but it means the protocol has no in-band caller-identity distinction beyond uid.                                                                                      |
| Tampering              | A malformed ndjson frame or a protocol-version-mismatched peer attempts to corrupt registry state or bypass request validation.                                                                 | Versioned handshake rejects a mismatched protocol version before serving a request (05 §UDS control plane); the contract-typed router validates every operation shape; every externally visible effect is journaled via 04's `appendEntry` (fsync-before-side-effect) before it takes effect (05 §Event bus).                                                      | None material beyond ordinary parser-robustness risk; the journal-before-effect discipline means even a successful tampering attempt leaves a durable, hash-chained record (04) rather than silently mutating state.                                                                                                                                                                                                                                                                                                              |
| Repudiation            | A caller denies having issued a sensitive operation (e.g., `run.cancel` on an in-progress verification run).                                                                                    | Every `run_transition`, `work_unit_transition`, `session_assignment`, and `adjudication_decision` this phase triggers is journaled pre-effect and typed against `JournalEntryType` (02, Gap 5; 05 §Event bus).                                                                                                                                                     | The journal records that _a_ same-uid peer issued the operation, not which of the two logical peer categories (CLI vs. gateway-forwarded) did so — the same uid-scoping limitation as Spoofing, reflected here as an attribution ceiling rather than an authentication gap.                                                                                                                                                                                                                                                       |
| Information disclosure | A same-uid process reads run registries, worker logs (via the log-streaming ring buffer), or session IDs it has no legitimate reason to see.                                                    | Socket mode `0600` inside a `0700` runtime dir prevents any other OS user's processes from connecting at all (05 §UDS control plane); this is the sole confidentiality boundary — the protocol has no further per-request authorization layer once a peer clears `SO_PEERCRED`.                                                                                    | Any same-uid process — not just the CLI and gateway — can read everything the router exposes, including `session_id` values (which double as `resume` bearer credentials, see Worker runtime §Spoofing below). This is the single-user trust model's structural ceiling, explicit in the design, not a defect to be patched within this phase.                                                                                                                                                                                    |
| Denial of service      | A slow subscriber to worker-log streaming stalls a worker's own output pipe, or a same-uid client floods the router with requests.                                                              | 1 MiB ring buffer per worker with backpressured subscribers — "a slow consumer never blocks the worker's own pipe; drops are counted, never silent" (05 §Log streaming, property-tested); idle resource budget (<100 MiB RSS, <1% of one core, 5 s heartbeats) is CI-measured with headroom, bounding baseline footprint (05 exit criteria).                       | No explicit rate limit or request-volume cap on a same-uid client's connection/request rate is named in 05's own text — a misbehaving (not necessarily malicious) same-uid client could still flood the router with requests faster than a well-behaved one; worker-process-level DoS is separately bounded by the SIGTERM→grace→SIGKILL ladder, but router-request-flooding itself is not enumerated as its own test scenario.                                                                                                   |
| Elevation of privilege | A caller invokes `run.*` or internal `worker.*` administration operations reserved conceptually for the gateway's forwarded surface, or reaches ChangeSet state through an undeclared shortcut. | `run.*` is UDS-only and "never registered as an MCP tool by this package, under any name" (05 §Router surface), closing one bypass route; there is no `change_set.*` MCP tool family anywhere in v1 (Gap 1) — ChangeSet state is reachable only through 11's `project.inspect`, itself reading this same UDS surface, giving one audited read path instead of two. | Because the trust boundary is uid-scoped rather than per-peer-scoped, the design deliberately does **not** distinguish "the CLI is allowed to do X but the gateway is not" within the protocol itself — both peer categories share the identical router and handler set by design ("one handler set, two transports," 05 §Goal). This is a flattening of privilege between the two named peer categories that is intentional, not accidental, but worth stating explicitly as a design property rather than an emergent property. |

## 2. Worker runtime

**Design:** per-work-unit Claude Code processes spawned via the Agent SDK's `query()` (confirmed v1 transport,
`docs/claude-code-adaptation.md` §5.3), implementing 03's `EngineAdapter` interface as `packages/engine-claude`
(06). Spawn mechanics — per-worker `HOME`/`TMP`/`CLAUDE_CONFIG_DIR` provisioning, the SIGTERM→grace→SIGKILL
termination ladder, and orphan reaping — are owned by the supervisor (05); the real journal-first
`AdjudicationCallback` policy, subscription-OAuth injection, `resume`/`forkSession` recovery, and `limitSignal`
detection are owned by 06. Confinement is layered defense-in-depth (`docs/claude-code-adaptation.md` §5.1):
compiled permission profile, per-call adjudication + journal, and OS sandbox, with the gateway as the sole
egress path for provider credentials.

| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Spoofing               | An attacker with journal or transcript-path access reuses a leaked `session_id` to `resume` another worker's conversation and continue acting in its context.    | Supervisor pre-generates and journals `session_id`/`SessionRef` (`session_assignment` entry) before the worker process starts (05/06); `resume` is scoped to the same project directory and its worktrees (`docs/claude-code-adaptation.md` §4.5); two concurrent same-project-dir sessions with distinct session IDs never interleave, confirmed live (`docs/engine-baseline.md` §7).                                                                                                                                                             | `session_id` itself functions as an implicit bearer credential for `resume` — no secondary check binds a `resume` call to "the same caller who owned the original session," only to "the same project directory." Access to a session_id is gated entirely by the UDS/journal/filesystem permission boundaries covered elsewhere (§1), not by a worker-runtime-local check.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Tampering              | A worker attempts to write outside its assigned worktree, or to weaken its own permission profile mid-run.                                                       | Compiled permission profile restricts `Edit`/`Write` to owned worktree paths (`Edit(//…/worktree/**)`, 03); OS sandbox denies filesystem write outside worktree+tmp and denies read of the control repo, journal, `~/.ssh`, `~/.aws` (`docs/claude-code-adaptation.md` §4.2, confirmed live via the `denyRead` ENOENT-masking behavior, `docs/engine-baseline.md` §6); every tool call is adjudicated and journaled _before_ it takes effect via `canUseTool` (06 §AdjudicationCallback), failing closed on a bridge crash or timeout, never open. | Sandbox network allowlisting is hostname-based without TLS termination by default (`docs/claude-code-adaptation.md` §10 risk 4 — domain-fronting caveat); mitigated only by workers having zero general egress under envelope default, not by the sandbox's own precision. compound-command (`&&`/`                                                                                                                                                                                                                                    |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            | `/`;`/`\|`) and process-wrapper (`nohup`/`timeout`/`nice`) smuggling are both denied because each subcommand must independently match an allow rule, confirmed live (`docs/engine-baseline.md`§3);`disableBypassPermissionsMode: "disable"`is always set (03), and no layer ever uses`bypassPermissions`/`--dangerously-skip-permissions` (`docs/claude-code-adaptation.md` §5.1). | Native worktree/subagent isolation is a young engine feature — a v2.1.203 bug let worktree-isolated subagents run in the parent checkout (`docs/claude-code-adaptation.md` §10 risk 6). Mitigated by never spawning write-capable workers into `.claude/worktrees/` or with `isolation: "worktree"` (06 §Spawn path: "never `.claude/worktrees/` or `isolation: "worktree"` — those stay manager-side only"), but the underlying engine feature's youth is a residual platform risk tracked only by the version-range gate, not eliminated. |

## 3. Envelope compiler

**Design:** a pure, property-tested function `compileEnvelope(envelope: AuthorizationEnvelope) →
CompiledWorkerProfile` (`03-envelope-compiler-engine-adapter.md`) turning an approved `AuthorizationEnvelope`
(02) into a permission profile, a sandbox profile, and two serializations (`WorkerSettingsJson`,
`WorkerSdkOptions`). Phase 03's own text names this the security keystone: "a defect here silently disables
enforcement for every worker in the system" (03 §Risks), and the phase runs a mandatory security-review pass
even though no runtime worker exists yet.

| STRIDE                 | Threat scenario                                                                                                                                                                                                                             | Mitigation (owning phase + mechanism)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                | Residual risk                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Spoofing               | An unapproved or forged `AuthorizationEnvelope` reaches the compiler and is compiled with full authority as if it had been through human approval.                                                                                          | `AuthorizationEnvelope` instances are minted only through 11's contract/DAG drafting flow and gated by 09's terminal-prompt HMAC token bound to the envelope's hash, journaled as `approval_token_mint` (09 §Approval UX foundation; `JournalEntryType`, 02).                                                                                                                                                                                                                                                                                                                                                                        | `compileEnvelope` itself performs no independent verification that its input was actually approval-token-bound — that binding is enforced entirely upstream (09/11), not re-checked at compile time. A bug in the call chain that passed an unapproved envelope into `compileEnvelope` would still produce a validly-shaped, but illegitimately authorized, profile; this is a call-chain risk, not a compiler-logic one.                                                         |
| Tampering              | A compiler defect silently emits a wider allow-list, drops a mandatory deny, or emits the Appendix B footgun (a blanket `mcp__*` deny that also kills the gateway's own allow).                                                             | Fast-check property suite (≥10k cases) proves "no allow outside the envelope; mandatory denies survive any envelope; no blanket `mcp__*` deny ever emitted" (03 exit criteria); mutation-test suite seeds deliberately broken compiler variants (blanket `mcp__*` deny, missing control-repo `denyRead`, space-before-colon `Bash` literal) and asserts each is caught by a failing test, not silently passed (03 §Security test plan); golden settings artifacts for three canonical envelopes are byte-stable across builds.                                                                                                       | Property and mutation tests catch the specifically seeded footgun classes; a genuinely novel emission bug outside that corpus is only caught if it also happens to violate one of the general invariants (no-allow-outside-envelope, denies-survive-any-envelope) — coverage is strong but the mutation corpus is finite by construction.                                                                                                                                         |
| Repudiation            | A dispute over which `AuthorizationEnvelope` produced a given compiled worker profile.                                                                                                                                                      | `session_assignment`/`adjudication_decision` journal entries tie a spawned worker's session back to the profile it launched under (04/05/06); the envelope's own approval is separately journaled (`approval_token_mint`, 09/11) — the causal chain envelope → approval → compile → spawn is journaled end to end even though `compileEnvelope` itself is a stateless pure function with no journal writes of its own.                                                                                                                                                                                                               | None material — the chain is fully reconstructable through the callers' own journaling, which is the intended design (03 explicitly does not write journal entries itself; it only "names which `JournalEntryType` member belongs at which lifecycle point," 03 §Out of scope).                                                                                                                                                                                                   |
| Information disclosure | A compiled profile embeds a literal secret value (rather than a masked placeholder) that later surfaces in a log, a committed golden artifact, or a worker's settings file.                                                                 | Sandbox profile emits `credentials.envVars` mask entries, never literal values (03 §Envelope compiler → sandbox profile); the compiler never handles literal secret values at all — the gateway (16) holds those, and workers only ever see the engine's own mask+`injectHosts` placeholder mechanism (`docs/claude-code-adaptation.md` §4.2).                                                                                                                                                                                                                                                                                       | Golden artifacts for the three canonical envelopes are committed to the repo for drift-testing; care must be taken that the fixtures used to generate them never embed real secret-shaped values. This is a fixture-hygiene residual for whoever authors the golden fixtures, not a defect in the compiler's own logic.                                                                                                                                                           |
| Denial of service      | A pathological or adversarial `AuthorizationEnvelope` shape causes the compiler to hang or degrade at scale (e.g., an extremely large allow-list or deeply nested path set).                                                                | `compileEnvelope` is specified as a pure, synchronous function (03 §Goal) with no I/O and no network calls; the ≥10k-case property suite exercises correctness invariants over a randomized input space, which incidentally exercises a broad shape distribution.                                                                                                                                                                                                                                                                                                                                                                    | No explicit performance/complexity bound against pathological envelope shapes (e.g., adversarially large allow-lists) is named in 03's test plan — the property suite targets correctness, not throughput or worst-case latency, so a scale-DoS specifically crafted around compiler performance is untested territory.                                                                                                                                                           |
| Elevation of privilege | The single highest-impact threat for this surface: any circumstance under which the compiler emits a profile granting capability beyond the approved envelope (write access outside the worktree, unintended egress, or subagent spawning). | Explicitly named the phase's own "security keystone" with a mandatory security-review pass required before the phase closes, even though no runtime worker exists yet to exploit a defect (03 §Risks); the adjudication-hook-bypass test specifically asserts a fake engine attempting a tool call with no `AdjudicationCallback` supplied, or one that throws, must fail closed, never open (03 §Security test plan); mandatory denies (`Agent`, `WebFetch`, `WebSearch`, `Bash(git push:*)`, `Bash(curl:*)`, `Bash(wget:*)`, plus control-repo/journal/credential paths) are proven to survive any envelope by the property suite. | Mitigation here is procedural rigor (mutation testing, property testing, mandatory review) rather than a structural impossibility — `docs/claude-code-adaptation.md` Appendix B's own verified footgun (a blanket `deny: ["mcp__*"]` silently killing the gateway allow, since deny always wins over allow at any level) demonstrates how subtle a single rule-interaction bug can be even with careful design; the mitigation reduces but does not eliminate this class of risk. |

## 4. Installer

**Design:** `engineering-orchestrator install/upgrade/uninstall` (`10-plugin-and-installer.md`) scaffolds a
target project's `CLAUDE.md` managed block, `.claude/settings.json` add-only keys (attribution off,
`enabledPlugins`), `.claude/agents/eo-*.md`, and a project-scope `.mcp.json` entry keyed
`GATEWAY_MCP_SERVER_NAME`, idempotently and reversibly, with an ownership/checksum/backup state store.
Distribution is via a SHA-pinned marketplace listing or a digest-pinned vendored `--plugin-dir`/`--plugin-url`
path.

| STRIDE                 | Threat scenario                                                                                                                                                                                        | Mitigation (owning phase + mechanism)                                                                                                                                                                                                                                                                                                                                                                                                                                | Residual risk                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Spoofing               | A malicious marketplace listing or vendored plugin source impersonates the legitimate orchestrator plugin.                                                                                             | Marketplace repo is SHA-pinned (`marketplace.json`); vendored `--plugin-dir`/`--plugin-url` installs are digest-pinned; the plugin's own `CapabilityManifest` entry is digest-pinned (02 schema, populated by 10 §Distribution).                                                                                                                                                                                                                                     | Pinning prevents silent post-trust substitution but does not itself vet initial content — that is 12's capability-quarantine job, and 10 explicitly has no dependency edge onto 12 at its own build time: "this phase has no dependency edge on 12 or 23... nothing here should assume quarantine has already run" (10 §Risks). The quarantine pass for the first-party plugin bundle happens at 23's release-gate stage, not as an installer-time gate.                                                                              |
| Tampering              | An external process mutates an installer-owned file between installs, or a process kill mid-upgrade leaves torn state.                                                                                 | Ownership/checksum/backup on-disk state store (10 §Installer artifacts); drift detector flags any single-byte external mutation of an owned file (10 exit criteria, `drift.fixtures` suite); upgrade backup + rollback + interrupted-upgrade recovery, explicitly tested against a kill-mid-write fixture (10 work item 5).                                                                                                                                          | Drift detection is reactive — surfaced by `doctor`, not prevented at the moment of tampering; a tampered file is not automatically rejected before the manager session next reads it, so a window exists between tampering and detection.                                                                                                                                                                                                                                                                                             |

## 5. Gateway

**Design:** the provider-neutral policy gateway (`16-gateway-core.md`), sole host of the single `eo_gateway`

## 7. Capability quarantine

**Design:** the fetch → pin → digest → verify-signature → SBOM/scan → sandboxed-test → human-approval pipeline
(`12-stack-detection-quarantine.md`) every executable capability — skills, plugins, hooks, MCP servers,
scanners — must pass before becoming a digest-pinned `CapabilityManifest` entry or a callable gateway tool.
The roadmap explicitly added this surface (and renderer) to the threat-model requirement to close gaps
flagged by phases 12 and 17, addressing `docs/claude-code-adaptation.md` §10 risk 11 (plugins/executable
capabilities as attack surface).

| STRIDE                 | Threat scenario                                                                                                                                                                               | Mitigation (owning phase + mechanism)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | Residual risk                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Spoofing               | A malicious package or plugin is fetched under a legitimate-looking name, or a digest is spoofed so a malicious artifact appears identical to a previously-approved one.                      | Fetch without credentials → pin immutable digest → verify signature/provenance where available (12 §Quarantine pipeline stages 1–3); the content-addressed capability store keys strictly on `(digest, permission footprint)`, property-tested so any change forces a different key (12 §Test plan Property); a changed digest or permission footprint forces re-audit (12 exit criteria).                                                                                                                                                         | Signature/provenance verification is only "where available" — for ecosystems without signed artifacts, the pipeline falls back to scan+sandboxed-test+human-approval alone; SLSA/CycloneDX outputs are explicitly "stored as evidence, not proof of benignity" (12 §Quarantine pipeline) — an intentionally honest, stated limitation rather than a hidden gap.                                                                                                                                                                                                                                                                               |
| Tampering              | A previously-approved capability is swapped post-pin for a malicious version, or the pipeline's own scanners (gitleaks/osv-scanner/Syft) are themselves compromised at bootstrap.             | Digest-pinning reproducibility is a conformance test — "two audits of a byte-identical candidate yield the identical digest" (12 §Conformance); "unsigned digest swapped post-pin" is an explicit seeded security-test scenario (12 §Security); scanners are bootstrapped through this same pipeline with vendored first-trust digests, and 12's own text applies `docs/claude-code-adaptation.md` §10 risk 11 to this bootstrap set explicitly (12 §Risks).                                                                                       | The scanner bootstrap is an acknowledged trust-root problem: the very first vendored digests for gitleaks/osv-scanner/Syft are trusted by documented procedure rather than verified by the pipeline itself — an unavoidable "turtles all the way down" limitation for any quarantine system, explicitly named rather than hidden in 12's own risk text.                                                                                                                                                                                                                                                                                       |

## Cross-surface residual themes

A few residual-risk patterns recur across more than one surface and are worth stating once, explicitly, rather
than only as scattered table cells:

1. **Same-uid trust flattening.** The UDS control plane (§1) and, transitively, everything built on top of it
   (worker session resumption via `session_id`, §2) share a single OS-user-account trust boundary with no
   finer-grained in-protocol caller distinction. This is a stated design choice, not an oversight, but it means
   the system's security model assumes the invoking OS account itself is not compromised — a foundational
   assumption worth stating plainly rather than leaving implicit.
2. **Procedural rather than structural mitigation at the two keystones.** Both explicit "security keystone"
   surfaces — the envelope compiler (§3) and the gateway (§5) — rely on mandatory security review plus
   property/mutation testing rather than a second independent technical control layered underneath their own
   logic. This is an appropriate response to the nature of the risk (a single compiled-configuration or
   plan-validation bug can matter more than throughput or edge-case correctness), but it means the quality of
   the review process itself is part of the security boundary, not just the code.
3. **The capability-quarantine journal gap is real and unresolved.** Phase 12's own text flags, and this review
   confirms independently, that capability-audit pass/fail/contradiction verdicts have no dedicated
   `JournalEntryType` member — only the subsequent approval-token mint is centrally journaled. `JournalEntryType`
   is deliberately closed at 13 members pending a coordinated resolution round (Gap 5); this is the single
   clearest "needs a follow-up decision" item this review identified that is not simply a residual-risk
   acceptance.
4. **Inbound content is generally less scrutinized than outbound content.** The renderer's lint pipeline (§8)
   and the connector layer's outbound rendering (§6) are both heavily hardened against what the system _emits_;
   what the system _reads back_ from a provider (Jira issue text, capability scan output) is bounded by size
   budgets and redaction but not by the same content-level scrutiny (Unicode attacks, secret patterns) applied
   to outbound artifacts. No surface in this design currently claims otherwise, so this is recorded as an
   observation, not a contradiction of any phase's own stated scope.
