EO_CRASH_FIXTURE_BROKEN=1, uses a genuinely unsafe append variant instead of
the real appendEntry: it open()s the segment file with "w" (TRUNCATE — an
immediate, irreversible destruction of the file's existing bytes, unlike
the real path's "a"/O_APPEND), then rewrites the whole file (prior valid
entries + one new line) in two unsynced halves with a sleep between them.
Killed at "after-open-file": the file is left EMPTY, losing both
previously-durable prior entries. Killed at "after-write": the file is left
containing only the first half of the concatenated content, an unparseable
or truncated-mid-entry mess that may also destroy earlier valid entries
(unlike a real torn TAIL write, which only ever affects the newest line).
