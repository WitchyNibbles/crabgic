# Phase 02 evidence

This directory is the evidence trail for `roadmap/02-contracts-and-schemas.md`'s
exit criteria, following the same convention `docs/evidence/phase-01/README.md`
established: each file name is prefixed `wiN-` (work item N) or
`exit-criteria-`, describes what it captures, and — where the phase's own
choreography calls for "failing-first" — a `*-failing.txt` file captured
**before** the fix/implementation landed, paired with a `*-passing.txt` (or
`*-passing-*.txt`/`*-clean.txt`) file captured **after**.

Five parallel workers built this phase's parts (contracts/schemas, state
machines, capability flags, config resolver, connector errors, renderer-core,
journal/gateway modules, the threat model doc); this worker (the integration
worker) then wired the public barrel, built the JSON Schema emitter, built
`packages/testkit`, fixed two cross-cutting defects, and ran the full repo
gate. Evidence files below are grouped by which worker/pass produced them —
the `wi1-{intake,execution,connector}-contracts-*`, `wi4-connector-errors-*`,
`wi5-journal-*`, `wi6-comms-renderer-*`, `wi7-gateway-constant-*`, and
`wi8-config-resolver-*` files were captured by the five parallel workers
during their own TDD passes; `wi1-json-schema-bytestable.txt`,
`wi10-testkit-{failing,passing}.txt`, and `exit-criteria-full-gate.txt` were
captured by this integration pass.

## Deviations

1. **The run-lifecycle (work item 2), `WorkUnitAttemptStatus` (work item 3), and `HighImpactCapabilityFlag` (work item 7's non-gateway half) modules were implemented in a prior session that was interrupted before their failing-first captures were saved.** Their passing suites — `packages/contracts/src/state-machines/run-lifecycle.test.ts` (incl. the ≥10k-case fast-check fuzz), `packages/contracts/src/state-machines/work-unit-attempt-status.test.ts` (incl. its own ≥10k-case fuzz), and `packages/contracts/src/capability-flags/high-impact-capability-flag.test.ts` — stand as evidence of correctness (all green, part of the 452 tests in `exit-criteria-full-gate.txt`), but no `*-failing.txt` counterpart exists for them because the RED state was never captured before the interruption. The failing-first protocol was followed for every module built after the interruption, including everything in this integration pass (`packages/testkit`, `docs/evidence/phase-02/wi10-testkit-*.txt`).
2. **Two defects were fixed at integration time, both now confirmed by this pass's own evidence:**
   - **`capability-manifest.test.ts`'s hand-typed `"eo_gateway"` literal.** `packages/contracts/src/contracts/capability-manifest.test.ts` (~line 20) hard-coded the string `"eo_gateway"` in its `mcp_server` fixture, tripping `gateway/server-name.test.ts`'s sole-definition-site scanner. Fixed by importing `GATEWAY_MCP_SERVER_NAME` from `../gateway/server-name.js` and using the constant instead of the literal. Confirmed passing in `exit-criteria-full-gate.txt` §5.
   - **This integration pass's own barrel (`packages/contracts/src/index.ts`) introduced a second violation of the same scanner** — an early draft of its doc comment quoted the literal `"eo_gateway"` inside a `//` comment (comments count: the scanner does a raw-text `.split()` over every `.ts` file, not an AST-aware scan). Caught by the full-suite run during this same integration pass and fixed by rephrasing the comment to reference the constant name only, never the literal string. This was never committed in a broken state; it is recorded here for transparency about what the full-gate run actually caught.
   - **Root `vitest.config.ts` had no `dist/**` exclude**, so compiled `dist/**/*.test.js` duplicates (left behind by any `tsc -b`) were picked up as real test files a second time. The fix required more than the one-line `test.exclude` addition first attempted: Vitest 4.1.10's `test.projects: ["packages/*"]` (a bare glob string) resolves each matched package directory with Vite's `configFile: false`, which — verified directly against `resolveTestProjectConfigs`/`initializeProject` in `node_modules/vitest/dist/chunks/cli-api.BK8pd4xc.js` — means that project **never loads or merges this root config's own `test` settings at all** (only a small fixed allowlist of CLI-only overrides crosses that boundary, and `exclude` is not among them). A bare `test.exclude: ["**/dist/**", ...]` addition was silently a no-op for every glob-matched sub-project. The working fix (see `vitest.config.ts`'s own doc comment) replaces the bare glob with an explicitly-enumerated array of `{ extends: true, test: { root, name } }` project objects — `extends: true` does load and merge the root config into each project, including `exclude` — while a small helper re-reads each package's own `package.json` `"name"` field so the per-project `|@eo/testkit|`-style labels are preserved (they are not free with `extends: true`, unlike with a bare glob). Confirmed: 452 tests / 37 test files (no `dist/*.test.js` duplicates anywhere) in `exit-criteria-full-gate.txt` §1, and the gateway scanner test (which depends on there being no stray file matches) passes in §5.
3. **`packages/testkit/package.json`'s `ajv`/`ajv-formats` were reclassified from `devDependencies` to `dependencies`, and `zod` was added as a `dependencies` entry.** `packages/testkit/src/ajv-harness.ts` — the integration harness this phase's Test plan says "03/16/18/19/20/22 import rather than re-deriving fixtures" — is a runtime module in this package's own public barrel; a workspace package importing `@eo/testkit`'s ajv harness needs `ajv`/`ajv-formats` resolvable as real runtime dependencies of `@eo/testkit`, not merely `@eo/testkit`'s own dev-time tooling. `zod` was added because `packages/testkit/src/fixtures/registry.ts` imports `type { z } from "zod"` directly (for the fixture registry's `z.ZodTypeAny` schema-field type) rather than only re-using zod values re-exported through `@eo/contracts`. `package-lock.json` was regenerated (`npm install`) to reflect this reclassification.
4. **`packages/contracts/package.json`'s `exports` map gained a `"./schemas/*.json"` subpath entry** (and `"files"` gained `"schemas"`), so `packages/testkit/src/ajv-harness.ts` can locate `packages/contracts/schemas/<kebabName>.json` via `import.meta.resolve("@eo/contracts/schemas/<kebabName>.json")` rather than hand-deriving a monorepo-relative filesystem path (which would be fragile outside this specific directory layout, and wouldn't work uniformly for every future consumer package at a different nesting depth).
5. **`PerformanceContract` fixture builder returns the `provisional` variant only.** `PerformanceContractSchema` is a discriminated union of `provisional`/`enforced` variants (11 populates the former at approval time; 15 builds the latter at gate time). To keep the testkit registry at exactly one builder per contract (matching the exit criterion's "all 21 contracts" framing) rather than two, `buildPerformanceContract()`'s default output is the `provisional` variant — documented in `packages/testkit/src/fixtures/performance-contract.ts`'s own doc comment. Both variants independently validate against the same `PerformanceContractSchema` union either way.
6. **A CJS/ESM type-interop workaround was needed for both `ajv` and `ajv-formats` in `packages/testkit/src/ajv-harness.ts`.** Under this repo's pinned `moduleResolution: "NodeNext"` + `verbatimModuleSyntax: true`, `import Ajv from "ajv"` type-resolves to the whole CJS module-namespace object rather than the constructable `Ajv` class (a known interop wrinkle for CJS-only packages with no `package.json` `"exports"` map) — worked around by importing the named `{ Ajv }` export instead (ajv's own `.d.ts` exports both a named and a default `Ajv`). `ajv-formats` only default-exports its plugin function, so no equivalent named-export escape hatch exists; that one required an explicit, documented, narrow `as unknown as (ajv: Ajv) => void` cast after a namespace import (`import * as ajvFormatsModule from "ajv-formats"`) — the runtime value is correct (verified by `ajv-harness.test.ts`'s own passing assertions); only the static type was wrong. Both workarounds are documented inline at their call sites.
7. **JSON Schema determinism settings** (`packages/contracts/scripts/build-schemas.ts`): `zodToJsonSchema(schema, { target: "jsonSchema7", $refStrategy: "none" })`, `JSON.stringify(schema, null, 2)` plus one trailing newline, and a fixed alphabetized-by-filename array (never `Object.keys` iteration order) driving emission order. `$refStrategy: "none"` fully inlines every schema (no `$ref`/`definitions` indirection) so each `schemas/*.json` file validates standalone via `ajv.compile()` with no second schema needing pre-registration — the same property `packages/testkit/src/ajv-harness.ts` relies on. `target: "jsonSchema7"` matches `ajv-formats`' `uuid`/`date-time` format keywords and ajv 8's default draft-07 dialect (no need for `ajv/dist/2019`).
8. **`packages/contracts/schemas/*.json` and `packages/testkit`'s generated build output were excluded from `prettier --check .`'s scope** (`.prettierignore` gained a `packages/contracts/schemas/` entry, mirroring the existing `package-lock.json` precedent) because the schema files' own byte-stability contract (item 7 above) is deliberately raw `JSON.stringify` output, not prettier's own JSON style — the two conventions differ (e.g. prettier collapses short arrays onto one line; `JSON.stringify(..., null, 2)` never does) and re-running `prettier --write` on the schemas would itself break the byte-stability exit criterion on the next `build:schemas` run. All source files this worker authored (`packages/testkit/src/**`, `packages/contracts/scripts/build-schemas.ts`, `packages/contracts/src/index.ts`) are prettier-clean with no ignore entries.
9. **`wi4-connector-errors-failing.txt`'s RED state is a module-resolution failure, not per-assertion failures.** The capture was taken with the implementation module (`connector-error.ts`) absent, so all it demonstrates is `Cannot find module './connector-error.js'` — the suite's individual assertions (member count, order, redaction behavior, round-trips) were never observed failing one by one against a stub implementation. This still proves tests-before-implementation ordering (the file's own note explains the type-level assertions cannot fail "harder" than the type not existing), but it is a weaker RED than an assertion-level capture would be: it cannot show that each assertion is individually capable of failing. Recorded here so future phases prefer stub-backed RED captures where practical.
10. **Formatting fixes applied to other workers' already-committed-equivalent files during the full-gate pass**, all whitespace/line-wrap-only (verified via `tsc -b`/`vitest run` staying green before and after): `docs/engine-baseline.md`, `docs/threat-model.md`, `packages/contracts/src/capability-flags/high-impact-capability-flag.test.ts`, `packages/contracts/src/state-machines/run-lifecycle.test.ts`, `packages/contracts/src/state-machines/run-lifecycle.ts`, `packages/contracts/src/state-machines/work-unit-attempt-status.ts`. This is the "you are the integrator" authorization from this worker's brief exercised narrowly — no semantic content changed in any of these files, confirmed by the full gate staying green (`exit-criteria-full-gate.txt`).
11. **Post-validation fix loop (2026-07-17): `CONNECTOR_ERROR_KINDS` member order corrected to roadmap order.** The Opus validation round found `packages/contracts/src/errors/connector-error.ts` ordering the union `…unsupported, ambiguous_write, transient, policy_blocked` while roadmap/02-contracts-and-schemas.md line 19 declares `…unsupported, transient, ambiguous_write, policy_blocked`; `connector-error.test.ts`'s "byte-matches the roadmap's declared member list, in order" test consequently asserted the wrong order under a truthful-sounding title. Fixed by swapping the two adjacent members in the source tuple, the test's expected array, and the test's `CONSTRUCTORS` table (set membership, and therefore all accept/reject behavior, unchanged). `schemas/remote-operation-record.json` embeds this enum's order, so `npm run build:schemas` was re-run twice (byte-stable, empty `diff -r` between consecutive builds) and the full gate re-run green — see `exit-criteria-full-gate.txt`'s dated ADDENDUM section for the capture.

## Design decisions recorded (within this phase's authority, not ambiguities)

- **Contracts barrel export strategy** (`packages/contracts/src/index.ts`): `export * from` per module, alphabetized within each category (shared primitives → state machines → capability flags → learning state → journal entry type → gateway constant → config resolver → connector errors → renderer-core → the 21 contracts alphabetically). A repo-wide export-name collision check (`grep`-based, across every non-barrel/non-test `.ts` file's top-level `export` declarations) found zero duplicate identifiers, so no explicit re-export renaming was needed anywhere.
- **`shared/schema-version-migration.demo.ts` is deliberately excluded from the barrel.** Its own doc comment states it is "deliberately NOT one of this phase's 21 real contracts" — a synthetic fixture pair proving out the schemaVersion+migration pattern, not a "shared primitive" in the sense the barrel re-exports.
- **Deterministic fixture-builder design**: every `build<Contract>()` function creates one fresh `FixtureContext` (a seed-`0` id provider + fixed-epoch clock provider) per call, so a single builder invocation can draw multiple distinct ids/timestamps internally while its *default* (no-overrides) output stays byte-identical across every run, on every machine — verified by `registry.test.ts`'s "every builder honors partial overrides immutably" test (`not.toBe` + `toEqual` across two successive calls).
