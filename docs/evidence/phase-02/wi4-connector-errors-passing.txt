=== Part 2: type-level enforcement (the actual security-exit-criterion check) ===

vitest transpiles TypeScript with esbuild and does NOT enforce
`@ts-expect-error` — the 3 type-level assertions in the "type-level
security" describe block above therefore execute successfully as plain JS
regardless of whether the suppressed line would actually fail to compile.
The real enforcement mechanism, as this phase's own work item 4 requires
("an `@ts-expect-error` line that stops erroring fails the build"), is
`npx tsc -b packages/contracts`.

First run, immediately after implementing connector-error.ts and before
relocating two `@ts-expect-error` comments onto the exact offending lines
(TypeScript attributes an excess-property error to the specific bad
property's line, not to the line where the object-literal assignment
starts) — this is the real, unedited failure this project surfaced:

Command:

  npx tsc -b packages/contracts

Output:

  packages/contracts/src/errors/connector-error.test.ts(147,5): error TS2578: Unused '@ts-expect-error' directive.
  packages/contracts/src/errors/connector-error.test.ts(156,7): error TS2353: Object literal may only specify known properties, and 'rawProviderBody' does not exist in type '{ message: string; kind: "validation" | "authentication" | "permission" | "not_found" | "conflict" | "rate_limited" | "unsupported" | "ambiguous_write" | "transient" | "policy_blocked"; provider: string; retryable: boolean; redactedDetail: string; }'.
  packages/contracts/src/errors/connector-error.test.ts(176,5): error TS2578: Unused '@ts-expect-error' directive.
  packages/contracts/src/errors/connector-error.test.ts(182,7): error TS2353: Object literal may only specify known properties, and 'rawProviderBody' does not exist in type 'ConnectorErrorInput'.

This is the exit criterion working exactly as designed: TS2353 proves
`rawProviderBody` is genuinely unconstructible on both the public
`ConnectorErrorData` type and the constructor's `ConnectorErrorInput` type;
TS2578 proves the `@ts-expect-error` comments were merely misplaced by one
property (moved to directly precede the `rawProviderBody: ...` line in each
case), not that the underlying type guarantee was ever missing.

Command after moving the two `@ts-expect-error` comments onto their exact
target lines:

  npx tsc -b packages/contracts --force

Output: (empty — exit code 0, zero diagnostics)

This confirms: (a) `rawProviderBody`/an unrecognized raw-body-shaped field
fails to type-check against `ConnectorErrorData` and against
`ConnectorErrorInput`; (b) `err.rawProviderResponse` fails to type-check
against a constructed `ConnectorError` instance; (c) every other line in
`packages/contracts` — the 6 connector contracts included — compiles clean
under the repo's strict TS config (`exactOptionalPropertyTypes`,
`noUncheckedIndexedAccess`, etc).
