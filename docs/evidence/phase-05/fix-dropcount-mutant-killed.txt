All three strengthened assertions independently catch the mutant:
1. ring-buffer-drops.test.ts's new deterministic "EXACT drop count" test (capacity 30,
   6-entry FIFO, subscriber falls 11 entries behind) — expected 11, mutant reports 1.
2. ring-buffer.property.test.ts's first property test — its new independent reference
   model (a brute-force loop-counted shadow simulation, not ring-buffer.ts's own
   closed-form subtraction) diverges from the mutant's fixed +1-per-poll count.
3. ring-buffer.property.test.ts's second property test ("never polls") — same
   independent reference model, exact match against a single large accumulated gap
   (pushCount up to 500 against a 512-byte/20-byte-line capacity).

Note: getting property test #1 to reliably catch this mutant required two supporting
fixes to the property test's own parameters (see fix-dropcount-mutant-restored.txt's
companion note and ring-buffer.property.test.ts's own comments): the action arbitrary's
poll-weighting was rebalanced to match its own doc comment ("1 = slow, rarely polled",
previously polled at the same rate as subscriber 0), and the buffer capacity/minLength
were rebalanced (capacityBytes 2048->250, minLength 1->60) after direct measurement
showed fast-check's default array-size heuristic clusters generated array lengths at
~1-12 elements even with maxLength: 200, so the previous capacity (large enough to
require 40+ pushes before any eviction) was rarely if ever exceeded in a typical run —
the property was checking almost nothing. Both fixes are pure test-quality
strengthening; no production code changed by them.
