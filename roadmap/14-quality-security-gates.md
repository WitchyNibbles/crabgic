## Goal

Before this phase, `verifying`/`final_verifying` are named run-lifecycle stages (02) with no enforcement behind them, and nothing binds a `Requirement` to proof it was met. After this phase, every work-unit attempt and every final-integrated candidate passes through a risk-tag-keyed, externally-extensible gate registry — TDD evidence, a monotonic coverage ratchet, stack-native test execution, SSDF-selected security scanning, and flake policy — each firing producing an `EvidenceRecord` (02) hash-bound to the exact object ID under test, so `Requirement → EvidenceRecord → exact object ID` resolves in both directions for any `ChangeSet`.

## In scope

- **Gate framework & registry:** an extensible, risk-tag-keyed gate registry — tags are `IntentContract`'s named sections (11: `scope/non-goals/audience/compatibility/security/performance/observability/rollout/acceptance`) plus this phase's own always-on default tags `tdd`, `coverage`, `flake` (no matching `IntentContract` section exists for these; they apply to every `ChangeSet` regardless of which sections are populated). `security` is shared: this phase's own SAST/secret/dependency scanners register under it alongside 21's connector-security fixtures; `performance` is used exclusively by 15's registered gate, nothing here contributes to it. Fires at the `verifying` (per-work-unit) and `final_verifying` (final-candidate) run-lifecycle stages (02). External phases register handlers with no new phase-dependency edge (same aggregation pattern as the gateway tool registry, Gap 1). Every firing — this phase's own or an external registrant's — emits one `EvidenceRecord` (command, exit status, env/toolchain fingerprint, timestamp, artifact digests, exact object ID).
- **TDD gate:** failing-test-first evidence — red at base journaled before implementation dispatch, green at candidate; a missing red-baseline `EvidenceRecord` for the same `Requirement` fails the gate.
- **Coverage:** ≥80% line+branch on greenfield projects; existing projects never regress below their recorded floor; changed instrumentable code reaches 80%; ratchet state journaled and monotonic.
- **Test execution:** unit/integration/E2E via stack-native commands declared on `ProjectProfile` (02), run inside the dispatching work unit's own worker-grade sandbox (03/06, provisioned by 13). Which adapter/scanner *categories* apply at all is decided from `StackEvidence` (02, populated by 12) — e.g. no JS-specific SAST ruleset fires without Node evidence; IaC adapters fire only when Terraform/CloudFormation files are detected. (`ProjectProfile` says *how* to run a stack's own tests; `StackEvidence` says *whether* a given gate category applies at all — the two schemas serve distinct roles here, not made explicit anywhere else in the roadmap.)
- **Security checks (SSDF-selected):** SAST (semgrep), secret scanning (gitleaks), dependency/license analysis (osv-scanner + policy); negative tests; fuzzing/DAST/container/IaC adapters where `StackEvidence` applies; CRITICAL/HIGH findings block. Scanner binaries resolve as digest-pinned entries from 12's content-addressed capability store — never fetched or executed ad hoc.
- **Root-cause policy detectors:** flag disabled checks, broad exception swallowing, hidden fallbacks appearing in a diff; configurable advisory→blocking; same normalized-finding shape as the security adapters.
- **Flake policy:** a rerun-then-pass result is marked `unstable` until fixed or explicitly quarantined; quarantine entries are journaled and expire on a recorded schedule, reverting to blocking.

## Out of scope

- **Stack detection, the capability-quarantine pipeline, and digest-pinning the scanner binaries themselves** (12) — this phase only consumes already-produced `StackEvidence` and already-pinned tool entries; it never runs a detector or a quarantine pass.
- **DAG dispatch, `TaskPacket` construction, worktree/sandbox provisioning, and model routing** (13) — this phase supplies gate definitions/thresholds that ride inside 13's `TaskPacket.gates` field and reads back results; it never spawns a worker or picks a model role.
- **`IntentContract`/`Requirement` authoring and section structure** (11) — this phase reads already-approved section names as its risk-tag vocabulary; it never assembles a contract.
- **Computing or freezing the base/integrated object ID, merge preflight, CAS ref updates** (07/08) — this phase gates against object IDs 07/08 already compute and freeze; it never touches Git plumbing.
- **Compiling worker permission/sandbox profiles or spawning the engine** (03/06) — gate commands execute inside whatever sandboxed worker session 13 already dispatches into.
- **`PerformanceContract` budget sourcing, the twin-worktree A/B methodology, its decision rules** (15) — this phase exposes the registry 15's gate registers into; it implements no performance-specific logic itself.
- **Authoring connector-specific security fixtures** — forged admin/delete, tenant boundary, redaction (21) — this phase only runs whatever 21 registers, inside the same framework.
- **Eval grading, shadow-run comparison, contamination detection** (22) — this phase supplies `EvidenceRecord`s/gate outcomes as grading ground truth; it never grades anything itself.
- **CLI command shape, argument parsing, output formatting for `evidence <change-set-id>`** (09) — this phase supplies the `EvidenceRecord` data that command surfaces; the command itself is 09's.
- **Provisioning disposable/live/containerized environments or release-scale orchestration** (23) — this phase builds gate logic once; 23 re-invokes the same `gates-conformance` job against the frozen release-candidate object ID.

## Interfaces produced

- **`packages/gates` extensible gate registry** — risk-tag-keyed (see In scope for the tag vocabulary); external registration requires no new phase-dependency edge. 15 registers its performance gate under `performance`; 21 registers connector-security fixtures under `security`. Fires at `verifying` and `final_verifying` (02).
- **`EvidenceRecord` (02 schema) emission** — this phase is the primary emitter: every gate firing produces one instance (command, exit status, env/toolchain fingerprint, timestamp, artifact digests, exact object ID), journaled as a `JournalEntryType.evidence_pointer` entry (02/Gap 5). This exact field shape is what 15 cites verbatim ("mirroring 14's own..."), and what 09's `evidence <change-set-id>` command, 21's traceability report, and 23's release-gate report all read.
- **Coverage ratchet store** — persists the monotonic coverage floor per project; consumed by 23 (re-verified, never re-designed, at release scale).
- **Flake-quarantine registry** — active/expired quarantine entries with expiry timestamps, built from journaled `evidence_pointer` history; consumed by 23 (re-run against the release candidate).
- **`gates-conformance` CI job** — the named, standalone-runnable seeded-fault matrix (mirrors 15's `perf-conformance` naming precedent). This is the exact artifact 23's "seeded-fault matrices from 14/15/22" bullet re-invokes for this phase.
- **Grading ground truth for 22** — this phase's `EvidenceRecord`/gate-outcome data is available for 22's dev/held-out eval grading. This confirms, from this phase's side, the architectural link 22's own text flags as an unconfirmed inference: `EvidenceRecord`'s existing command/exit-status/object-ID shape (above) is sufficient as eval ground truth; no interface beyond the `EvidenceRecord` emission already described here is needed.

## Interfaces consumed

- **From 12** (`packages/detect`, direct dependency — 12's own header: "Unlocks: 14 (gate selection)"): `StackEvidence` (02 schema, populated by 12) — drives adapter/gate-category selection. Content-addressed capability store (`$XDG_CACHE_HOME/engineering-orchestrator/<project-hash>/capability-store/`, pinned in 04) — this phase's scanner binaries (semgrep, gitleaks, osv-scanner, Syft where SBOM applies) resolve as digest-pinned entries from it ("tools digest-pinned via 12" — this phase's own In-scope text); a tool whose resolved digest no longer matches fails the gate closed rather than running a stale/tampered binary.
- **From 13** (`packages/scheduler`, direct dependency): `TaskPacket` dispatch — per-work-unit gate execution (TDD/coverage/stack-native tests) rides inside the same worker session/`TaskPacket` as the work unit's own implementation, via 13's `TaskPacket.gates` field (13 §In scope) carrying this phase's definitions/thresholds; final-candidate re-verification (work item 6) dispatches as its own `TaskPacket` through 13's executor — 13's own model-routing policy applies, this phase introduces no new role. Artifact store — raw tool output (coverage lcov files, SAST/secret-scan JSON reports) stored as bounded artifacts; this phase's `EvidenceRecord.artifactDigests` reference them, never inlining raw output. Attempt policy — a gate failure is the "new diagnostic evidence" 13's repair-attempt policy (one initial + two evidence-driven repairs) requires before a repair dispatch; this is a journal-checked evidentiary requirement only, never a claim that repair evidence must be rendered/lint-passed text (17's renderer is out of scope for this relationship entirely — see Gap 6 rationale).
- **From 11** (transitively via 13, through `Requirement` IDs riding on `ChangeSet`/`WorkUnit` — no new dependency edge, since 13 already depends on 11; 11's own text already names this: "`Requirement` IDs consumed by 14 (requirement → evidence resolution)"): stable `Requirement` IDs — this phase's `EvidenceRecord`s resolve bidirectionally against them.
- **Ambient, via `packages/contracts` (02)** (same convention every phase uses, not a direct Depends-on edge): `EvidenceRecord`, `ProjectProfile`, `WorkUnitAttemptStatus` (Gap 4 — this phase's gate verdicts are the evidentiary basis 05/13 use to drive an attempt's `succeeded`/`failed` status; this phase never transitions the enum itself), `JournalEntryType` member `evidence_pointer` (Gap 5), run-lifecycle states `verifying`/`final_verifying`; `packages/testkit` fixture builders (fake engine, fake `TaskPacket`s) for gate-framework fixtures.

## Work items

1. Gate framework core: registry (register/list/fire-by-tag), the risk-tag vocabulary above, `EvidenceRecord` emission, journaled as `evidence_pointer`. Failing-first: a stub gate registers and fires through the registry, before any real check (TDD/coverage/security) exists.
2. TDD-evidence gate, cooperating with 13's scheduler dispatch and repair-attempt policy. Failing-first: a fixture attempt missing a red-baseline `EvidenceRecord` for its `Requirement` is rejected.
3. Coverage adapters (lcov/istanbul, go cover, pytest-cov) + ratchet store, with `StackEvidence`-driven adapter selection. Failing-first: one parser fixture per adapter at a known percentage; a ratchet-regression fixture (recorded floor 82% → new run 79%) blocks.
4. Security adapters (semgrep, gitleaks, osv-scanner) with normalized findings + CRITICAL/HIGH-blocks policy, plus root-cause policy detectors; tool binaries resolved via 12's capability store. Failing-first: one seeded-finding fixture per adapter, red before the adapter exists.
5. Flake detector + quarantine registry with expiry, built on the registry/`EvidenceRecord` primitives from item 1. Failing-first: a scripted flaky-then-passing fixture is marked `unstable`, never silently green.
6. Final-candidate orchestration: re-fire the full registered gate set (this phase's own plus any external registrants) against the exact integrated object ID (08), dispatched as its own `TaskPacket` through 13's executor. Failing-first: fake-engine E2E where a per-work-unit gate already passed, but only the rerun against the truly-integrated object ID is asserted as the actual pass condition.

## Test plan

- **Unit:** per-adapter coverage-parser fixtures (lcov/istanbul, go cover, pytest-cov) against known-percentage fixtures; `EvidenceRecord` schema round-trip per gate type; ratchet-regression fixture blocks.
- **Property:** fast-check over randomized coverage-history sequences — the ratchet floor is monotonic non-decreasing regardless of insertion order; fast-check over randomized gate-registration order — dispatch is tag-keyed, never order-dependent.
- **Integration:** scheduler dispatch of a gate-bearing `TaskPacket` on the fake engine (`packages/testkit`); final-candidate re-verification fixture — a per-work-unit gate that already passed is re-fired against the truly-integrated object ID and asserted as the actual pass condition (proves final-candidate re-verification never trusts a cached per-work-unit result); `StackEvidence`-driven selection fixture — a pure-Go fixture repo never invokes a JS-specific SAST ruleset.
- **Conformance:** registry-extensibility test — a stub external gate registers and fires at `final_verifying` with zero code change inside `packages/gates` (the mechanism 15/21 rely on); every emitted `EvidenceRecord` round-trips through 02's `JournalEntryType` discriminated-union exhaustiveness check (shared CI job with 02/04, mirroring 22's identical exit criterion).
- **Security:** planted secret (AWS-shaped test key in a fixture file) → gitleaks fixture blocks; known-CVE test double in a fixture manifest → osv-scanner fixture blocks; an intentionally vulnerable pattern in a fixture source file → semgrep fixture blocks; a disabled-check diff (fixture commenting out an assertion / widening to a bare except) → root-cause detector flags; a scanner binary whose digest no longer matches 12's pinned entry → resolution fails closed rather than running a stale/tampered binary (mirrors 12's own unsigned-digest-swap vector).

## Exit criteria

- [ ] `gates-conformance` CI job blocks correctly on all six seeded fixtures independently: coverage regression, flaky-then-passing test, planted secret, vulnerable dependency, SAST finding, disabled-check diff.
- [ ] Coverage ratchet floor is monotonic non-decreasing across a simulated multi-`ChangeSet` history (fast-check property report, CI-archived).
- [ ] `Requirement → EvidenceRecord → exact object ID` resolves in both directions for a fixture `ChangeSet` (named integration test).
- [ ] A rerun-until-pass result is marked `unstable`; an explicit quarantine suppresses blocking until its recorded expiry, after which it reverts to blocking (flake-registry expiry test).
- [ ] TDD gate rejects a fixture attempt lacking a red-before-green `EvidenceRecord` pair for the same `Requirement`.
- [ ] A stub external gate registers into the registry and fires at `final_verifying` with zero code change inside `packages/gates` (registry-extensibility conformance test).
- [ ] Every emitted `EvidenceRecord` round-trips as a `JournalEntryType.evidence_pointer` entry through 02's discriminated-union exhaustiveness check.
- [ ] Final-candidate re-verification fires against the truly-integrated object ID rather than trusting a cached per-work-unit result (integration test with a deliberately stale per-work-unit pass).

## Risks & open questions

- **`StackEvidence` vs `ProjectProfile` (resolves 12's flagged gap):** 12's own Risks section flags that neither 14 nor 15 names `StackEvidence` verbatim, leaving the `ProjectProfile`↔`StackEvidence` relationship unstated. This file's In-scope "Test execution" bullet is the first to make it explicit: `ProjectProfile` supplies *how* to run a stack's own commands; `StackEvidence` supplies *whether* a gate category applies at all. Flagging so 15's author and the reconciler can adopt the same split rather than re-deriving it.
- **Risk-tag vocabulary is a design inference, not stated verbatim upstream:** tying registry tags to `IntentContract`'s named sections (plus this phase's own `tdd`/`coverage`/`flake` defaults) is this phase's own read of 11's section list and 15's "under the IntentContract's existing `performance` section tag" phrasing — internally consistent with both, but not itself asserted anywhere before this file.
- **No dedicated `JournalEntryType` member for gate/flake evidence, by design:** all of this phase's journal writes map onto the existing `evidence_pointer` member (02/Gap 5's union is closed at exactly 13 members — Gap 5's own rationale rejected a 14th). This mirrors 12's identical treatment of `capability.audit` decisions; not an oversight.
- **Adaptation §10 risk 9 (subscription-auth, token-based accounting):** final-candidate re-verification is itself a worker dispatch consuming turns under the owner's subscription, not a metered budget — track token/turn usage from the result `usage` block, never `--max-budget-usd`, mirroring 15's and 22's identical treatment.
- **No new unconfirmed Claude Code engine facts:** gate execution inherits its worker sandbox, dispatch, and session routing entirely from already-resolved 03/06/13 work; nothing here needs a `docs/engine-baseline.md` citation (same posture as 15/22's own Risk notes).
- **Gap 6, confirmed no-op for this phase:** the binding resolutions explicitly rejected tying this phase's repair-dispatch "new diagnostic evidence" requirement to 17's rendered/lint-passed review-comment template — that evidence is journal-checked only (see Interfaces consumed → 13). This file introduces no relationship to `packages/renderer` (17).
