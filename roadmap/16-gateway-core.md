## Goal

The provider-neutral policy gateway and the sole host of the single `eo_gateway` MCP server (name owned by 02): secure transport, secret references, capability snapshots, and a plan → validate → journal-before-network-I/O → apply → read-back → verify → record mutation pipeline with exactly-once semantics, exposed to the manager and to every worker through one extensible tool-registration API — tested throughout against scriptable fake providers. This phase natively implements `tracker.*`, `observability.*`, `evidence.get`, `evidence.attach`, and `result.submit`, and forwards `run.status`/`run.cancel` over UDS to 05's own router rather than reimplementing them; 09's `gateway mcp` command is a thin argv shim that boots this phase's server over stdio, and 11/12 later register their own already-built handlers (`project.inspect`/`contract.approve`, `capability.audit`/`capability.approve`) into the same registry, each with no new cross-phase dependency edge. Before this phase lands, 09 ships `gateway mcp` against a stub tool list and no connector has a transport to build against; after it, the registry is real, every mutating tool is exactly-once and SSRF-hardened, and 18/20 have the transport their Jira/Grafana resource clients are built on, not beside.

## In scope

- **ExternalConnection store:** provider, deployment type, exact HTTPS base URL, allowed redirect origins, tenant/org/project/folder allowlists, custom CA reference, allowed resources/actions, discovery TTL; secret **references** only (env, file 0600, exec backends; extensible) — never a literal credential in worker- or manager-reachable state.
- **Transport security:** TLS verification, redirect revalidation before credentials attach, scheme/origin/IP-range/base-path allowlists (SSRF guard incl. private ranges + DNS pinning), ≤4 in-flight per connection, Retry-After + jittered bounded backoff, write serialization per tenant+resource, O(page) pagination, N+1 detection.
- **CapabilitySnapshot:** product/edition/version/API families/resources/actions/permissions/expiry; 15-min cache; invalidation on auth/permission/unsupported errors; unknown versions read-only.
- **Mutation pipeline:** `RemoteMutationPlan` (connection+tenant, canonical target, action+capability, redacted diff, desired-state hash, idempotency key, expected remote revision, impact+rollback class, envelope ref) → persist `RemoteOperationRecord` **before network I/O**, via 04's idempotency registry → apply → read-back compare → verify → record. Same-ID+same-content replays the recorded result byte-identical; different content is a typed conflict (04's registry), never a silent overwrite.
- **Provider dispatch:** a provider-keyed extension point inside `tracker.*`/`observability.*` (keyed on `ExternalConnection.provider`), distinct from the MCP tool registry below — 18's `JiraResourceClient` and 20's `GrafanaProviderAdapter` register into it, and adding a provider never adds a new tool name; 18/19/20 register no MCP tool of their own.
- **Ambiguity:** marker-reconciliation interface declared here, adapters implement it (Jira entity properties, 18; deterministic UIDs + annotation tags, 20); ambiguous POST timeout → search marker before retry; unresolvable → canonical `ambiguous_write`, outcome treated as unknown, **block** — never a guessed duplicate.
- **Retry ladder:** GET free within budget; PUT/PATCH deterministic + precondition only; POST never blind; 409/412 → fetch, rebase-or-block.
- **Budgets:** 32 KiB item / 256 KiB result enforced here (typed truncation errors) — deliberately independent of the unconfirmed engine-side `MAX_MCP_OUTPUT_TOKENS` (adaptation §7 point 2, §10 item 10).
- **Sole MCP host & extensible tool-registration API:** this phase hosts the one `eo_gateway` MCP server (`GATEWAY_MCP_SERVER_NAME`, constant owned by 02) and natively registers `tracker.search/get/plan_create/plan_update/plan_transition/plan_comment/apply`, `observability.search/get/query/plan_create/plan_update/apply`, `evidence.get`, `evidence.attach`, `result.submit`, and forwards `run.status`/`run.cancel` over UDS to 05's own contract-typed router — never a second implementation of those two. The same registration API, not just these native families, is what 11 (`project.inspect`, `contract.approve`) and 12 (`capability.audit`, `capability.approve`) plug their own already-built handlers into when they land, with no new cross-phase dependency edge. There is **no** `change_set.*` family (`project.inspect`, 11, answers ChangeSet-state queries instead) and **no** `learning.*` family (promotion/review is CLI-only, 22); this registry must never grow either one (22's grep-based CI gate scans this package's registered tool names for exactly that). Every mutating tool requires run ID, idempotency key, expected state version, envelope reference, and a validated plan.
- **Upstream MCP client (optional, flag-gated):** for a provider that also exposes its own official MCP server (Atlassian, Grafana), this phase — never a worker, and never the manager directly — is the one process permitted to act as MCP *client* to it, behind a per-connection capability flag with REST remaining primary and default; adaptation §7 point 1 is unconditional on the other side of that boundary: no worker ever sees an upstream provider's MCP server, only this phase's own `tracker.*`/`observability.*` tools.
- **Fake providers (testkit):** scriptable tracker + observability doubles with fault injection (429/401/409/412, malformed pages, mid-POST timeouts) — extended, not reimplemented, by 18/19/20's own fault matrices.

## Out of scope

- New MCP tool families beyond the ones this phase natively registers — 11 (`project.inspect`, `contract.approve`) and 12 (`capability.audit`, `capability.approve`) own and register their own handlers into this phase's registry; this phase hosts the registry, not their business logic.
- The `gateway mcp` CLI-level argv shim — parser, help text, stdio process bootstrap, doctor integration — 09; this phase supplies the server 09's shim boots, never the shim itself.
- The `connection add|list|doctor|capabilities` command's parser/argv surface — 09 (ships it `NOT_IMPLEMENTED` until wired); this phase supplies the `ExternalConnection` store and capability-snapshot logic that backend calls into, never the command surface.
- Provider-specific resource semantics: discovery field mapping, workflow-transition mapping, milestone sync, OAuth/PAT token management — 18 (Jira Cloud), 19 (Jira Data Center), 20 (Grafana); this phase supplies only the generic transport, pipeline, and provider-dispatch point they plug into.
- Deciding whether a given deployment enables the optional upstream-MCP-client wrap for a provider — that per-connection flag lives in the `ExternalConnection` config 18/20 populate (or a human operator sets), never decided unilaterally by this phase.
- Rendering or linting of any communication text that travels through a connector write (ADF, PR/review-comment text, Grafana annotation templates) — 17; this phase moves already-rendered bytes, never authors or lints them.
- Approval-preview rendering of planned remote mutations, and `IntentContract`/`AuthorizationEnvelope` assembly — 11; this phase executes an already-approved plan, it never decides whether one should be approved.
- Binding confirmed remote revisions onto `EvidenceRecord`, requirement↔resource traceability views, and drift-CI replay — 21; this phase only produces the confirmed revision its own read-back step yields.
- `docs/threat-model.md` authorship — 02; this phase's transport/secret/pipeline surface is one of the document's required review passes, not the document itself.
- A `change_set.*` or `learning.*` MCP tool family, under any name, ever — deliberately absent everywhere in v1 (Gap 1 resolution; see In scope).

## Interfaces produced

- **`ExternalConnection` store** (02 schema; this phase implements the store) — CRUD + secret-reference resolution over env/file-0600/exec backends. Consumed by 09 (`connection add/list/doctor/capabilities` backend), 18, 19, 20 (populate/read connections for their own providers).
- **Connection-doctor reachability probe** — a primitive that exercises a stored connection end-to-end (incl. custom-CA validation) without performing a mutating call. Consumed by 18/19/20's own doctor-check functions (19's own words: "connection-reachability probe exercising 16's custom-CA path") and surfaced to the human through 09's `connection doctor <id>`.
- **Transport security stack** (TLS + redirect revalidation + SSRF guard + throttle/backoff + per-tenant+resource write serialization + O(page) pagination + N+1 detection) — consumed by 18 ("this phase's resource clients are built on this transport, not beside it," 18's own words), 19 (reuses 18's cross-worker throttling), 20.
- **`CapabilitySnapshot` cache** (02 schema; this phase owns the cache, 15-min TTL, and invalidation rules) — consumed by 18, 20 (populate instances from provider discovery); unknown versions default read-only.
- **Mutation pipeline** (`RemoteMutationPlan` → `RemoteOperationRecord` persisted before network I/O via 04's `IdempotencyRegistry` → apply → read-back → verify → record) — consumed by 18, 20 (every create/update/transition/comment call submits through this; neither phase persists `RemoteOperationRecord` itself), 21 (binds the confirmed revision this pipeline's read-back step yields onto `EvidenceRecord`).
- **Marker-reconciliation interface** — declared here, implemented by 18 (Jira entity properties) and 20 (deterministic UIDs + annotation tags); exercised by this phase's own ambiguous-POST retry search and by 21/23's exactly-once E2E scenarios.
- **Canonical-error redacting constructors** (02's 10-member union) — consumed by 18, 19, 20 (map every provider failure to exactly one member; no raw body crosses the boundary), 21 (`ambiguous_write`/`unsupported` block `final_verifying`).
- **Provider-dispatch point** (provider-keyed extension inside `tracker.*`/`observability.*`) — consumed by 18's `JiraResourceClient` and 20's `GrafanaProviderAdapter`; adding a provider never adds a tool name.
- **Result/item size budgets** (32 KiB item / 256 KiB result, typed truncation errors) — consumed by 18, 19, 20 (query/aggregation layers stay under this).
- **The `eo_gateway` MCP server + its extensible tool-registration API** (`GATEWAY_MCP_SERVER_NAME`, 02) — natively registers `tracker.*`, `observability.*`, `evidence.get`, `evidence.attach`, `result.submit`, and forwards `run.status`/`run.cancel` to 05; wire names `mcp__${GATEWAY_MCP_SERVER_NAME}__<family>.<op>` (e.g. `mcp__eo_gateway__tracker.search`, `mcp__eo_gateway__result.submit`). Consumed by 09 (its `gateway mcp` argv shim boots this server over stdio — 09 owns no tool behavior), 11 and 12 (register `project.inspect`/`contract.approve` and `capability.audit`/`capability.approve` into it at their own build time, no new dependency edge), 23 (full-registry completeness release gate).
- **Fake-provider testkit harness** (scriptable tracker + observability doubles, fault injection: 429/401/409/412, malformed pages, mid-POST timeouts) — built in work item 6 (20's own text cites this exactly: "built by 16 work item 6"), directly extended by 18 ("extends 16's harness," 18's own words) and 20; extended transitively by 19, which builds on 18's already-extended fake-Jira rather than on this phase's harness directly.
- **`docs/threat-model.md` review obligation** — this phase's transport/secret-handling/pipeline surface is a required review pass against that document (02's own citation of this phase).

## Interfaces consumed

**From 02 (`packages/contracts`):**
- Schemas this phase implements logic around, without redefining: `ExternalConnection`, `CapabilitySnapshot`, `RemoteMutationPlan`, `RemoteOperationRecord`. (`RemoteResource` is a related 02 schema, but 02's own consumer table names only 18, 20, and 21 for it, not this phase — this phase's pipeline treats a mutation's canonical target/read-back state as opaque bytes; 18/20 are the ones who construct `RemoteResource` instances from it.)
- Canonical connector-error union (10 members) — this phase's redacting constructors instantiate it; no raw provider-body field exists on the public type by construction (02's own type-level guarantee).
- `GATEWAY_MCP_SERVER_NAME` constant (value `"eo_gateway"`) — this phase's SDK server registration and every wire-level `mcp__${GATEWAY_MCP_SERVER_NAME}__<tool>` name derive from this; never a second hand-typed literal (02's repo-wide grep/golden-value CI check enforces this across `packages/*`).
- `JournalEntryType`'s `remote_operation_record` member — every `RemoteOperationRecord` this phase persists before network I/O is journaled against exactly this member.
- Run-lifecycle enum — canonical `ambiguous_write` routes toward the run's `blocked` terminal (02's own disambiguation: a different enum than Jira's own 4-state ticket-status `blocked`, 18, sharing only the token).
- Testkit fixture builders — this phase's fake providers are built against these rather than inventing separate fixture plumbing.

**From 04 (`packages/journal`):**
- `IdempotencyRegistry.checkOrRecord(operationId, contentHash, compute)` → `{ status: "replayed" | "recorded" | "conflict", result? }` — the exactly-once primitive the mutation pipeline calls before any network I/O; 04 supplies only the `(operationId, contentHash)` primitive, this phase owns the plan→validate→journal→apply→read-back→verify→record sequence wrapped around it.
- `appendEntry`/`queryEntries` — the durable write/read path every `RemoteOperationRecord` flows through.
- `runKillHarness` — reused directly for this phase's own crash-before/after-remote-commit exactly-once matrix (04 already names 05/13/23 as reusers; this phase is another).

**From 05 (`packages/supervisor`):**
- Contract-typed router ops `run.status`/`run.cancel` (05's own "Contract-typed router (`run.status`, `run.cancel`, internal `worker.*`)") — this phase's MCP-visible `run.status`/`run.cancel` tools are forwards over UDS to these, never a second implementation.
- UDS API (ndjson request/response, socket 0600 in 0700 dir, `SO_PEERCRED` uid check) — the transport those forwards ride over; 05's own trust-boundary description names the gateway as one of the two callers admitted to the socket, alongside the CLI.

## Work items

1. Connection store + secret-reference resolution (env, file 0600, exec backends; extensible).
2. HTTP client wrapper with the full security/throttle/serialization stack (TLS/redirect revalidation/SSRF guard/backoff/per-tenant+resource serialization/pagination/N+1 detection).
3. `CapabilitySnapshot` cache + invalidation rules.
4. Mutation pipeline (`RemoteMutationPlan` → journal-before-network-I/O via 04's idempotency registry → apply → read-back → verify → record) + reconciliation interfaces + canonical-error mapping (provider bodies never leak) + 32 KiB/256 KiB budget enforcement with typed truncation errors.
5. `eo_gateway` MCP server + extensible tool-registration API + provider-dispatch point: native `tracker.*`/`observability.*`/`evidence.get`/`evidence.attach`/`result.submit` handlers wired to items 1–4, forwarded `run.status`/`run.cancel` to 05, the registry surface 11/12 later populate, and the provider-keyed extension point inside `tracker.*`/`observability.*` that 18/20 register their resource clients into. Failing-first: booting against an empty native-handler set still lists a well-formed, empty registry without crashing — mirroring 09's own stub-registry convention — before any native handler is wired; a duplicate-name registration attempt is rejected before the check exists.
6. Fake providers (testkit) + fault matrix (429/401/409/412, malformed pages, mid-POST timeouts), extending 03's fake-engine fixture format — the harness 18 extends directly and 20 extends independently (19 extends 18's already-extended copy).

## Test plan

- **Unit:** secret-reference resolver per backend (env/file-0600/exec); redirect/SSRF allowlist matcher (foreign origin, private IP range, scheme mismatch); canonical-error constructors reject a raw-body field at the type level; retry-ladder decision table per verb.
- **Property (fast-check):** random redirect chains never reach a private/foreign-origin target pre-credential-attach; random `(operationId, contentHash)` sequences never silently overwrite; random paginated fixtures keep memory O(page) regardless of total item count; random ambiguous-POST timing never produces a duplicate remote object.
- **Integration:** fake-provider matrix (tracker + observability) across the full fault set (429/401/409/412, malformed pages, mid-POST timeouts); `gateway mcp` boot listing exactly the native tool set over stdio to a stub MCP client, plus a test-registered extra tool proving the registration API is genuinely extensible, not hardcoded to the native families.
- **Conformance:** exactly-once crash matrix reusing 04's `runKillHarness` — kill before/after the remote-commit point, identical-content replay, changed-content rejection, ambiguous-POST reconciled-or-blocked — none produce a duplicate or a silent overwrite; provider-body redaction proven by a live substring search across every error/log/artifact surface.
- **Security:** SSRF suite (foreign-origin redirect, private-IP redirect, scheme downgrade — all refused pre-credential-attach); custom CA honored against a disposable self-signed test server; forged delete/admin/impersonation calls fail before any network I/O; a blanket `mcp__*` deny is never required to keep this server's tools safe (Appendix B's own footgun, exercised here from the gateway side).

## Exit criteria

- [ ] Exactly-once matrix on fakes: crash before/after remote commit, identical replay, changed-payload rejection, ambiguous POST reconciled or blocked — none produce a duplicate or silent overwrite. Evidence: `runKillHarness`-driven CI suite.
- [ ] SSRF suite: foreign-origin/private-IP redirects refused pre-credentials; custom CA honored against a disposable self-signed server.
- [ ] Retry ladder proven per verb (GET free, PUT/PATCH deterministic+precondition, POST never blind, 409/412 fetch-rebase-or-block); per-resource write order preserved under concurrency.
- [ ] Budgets enforced (32 KiB item / 256 KiB result, typed truncation errors); pagination memory stays O(page) on a 10k-item fake.
- [ ] Leak hunt: no raw provider body in any error, log, or artifact (live substring search).
- [ ] `gateway mcp` lists exactly this phase's native tool set (`tracker.*`, `observability.*`, `evidence.get`, `evidence.attach`, `result.submit`, forwarded `run.status`/`run.cancel`) over stdio to a stub MCP client, and accepts one externally-registered tool with no name collision — proving the registration API 11/12 depend on later is genuinely open, not hardcoded.
- [ ] No `change_set.*` or `learning.*` name is ever registered — repo-wide grep/golden-value check over `packages/gateway`'s registered tool names (the substrate 22's own CI gate scans).
- [ ] Every SDK server registration and wire-level tool name references `GATEWAY_MCP_SERVER_NAME` — zero hand-typed `"eo_gateway"` literals anywhere in `packages/gateway` (02's repo-wide grep gate).
- [ ] Provider-dispatch point resolves to the correct provider-specific client for a fixture `ExternalConnection` per registered provider; an unrecognized provider value is rejected before any network call.
- [ ] Connection-doctor reachability probe (incl. custom-CA validation) succeeds against a disposable fixture connection and fails informatively against an unreachable one, exercised directly against this package, ahead of 18/19/20's own doctor-check wiring.
- [ ] When the optional upstream-MCP-client wrap is enabled for a fixture connection, no additional MCP server ever appears in a simulated worker's `mcpServers` config — only this phase's own native tools are worker-visible.
- [ ] Security review sign-off recorded against `docs/threat-model.md`.

## Risks & open questions
